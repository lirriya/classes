Architecture and application of Web

Une application web

C'est une application web est un logiciel accessible via un navigateur web. Elle fonctionne grace à une architecture répartie entre différents composants

  • Navigateur web(client): Interface utiliseé par l'utilisateur. Envoie des requetes (souvent HTTP) au serveur
  • Serveur web: Recoit les requetes, Traite les donneés et renvoie les réponses (pages HTML, JSON, etc.)
  • Serveur de base de données (SGBD): stocke et gère les donneés
  • Serveur d'authetification: vérifie l'identité de l'utilisateurs, souvent via LDAP

Architechture

On utilise l'architecture client/serveur

Déroulement:

  1. Le client (navigateur) envoie une requete au serveur.
  2. Le serveur traite la requete et envoie une réponse (souvent une page web ou des données)
  3. Le serveur peut aussi agir en tant que client pour faire appel à un autre serveur (ex. une API, une base de données, etc)

HTTPD

httpd = HyperText Transfer Protocol Daemon

C'est un programme serveur(souvent Apache ou Nginx) qui attend et traite les requetes HTTP venants des clients

Il tourne en tache de fond sur le serveur

Son role: recevoir des requetes HTTP(comme GET, POST), les analyser et répondre

Que fait le client quand il clique sur un lien HTML?

"a href="http://www.iro.umontreal.ca/~vaucher/test.html">Fichier /a"

Cela déclanche un requete GET comprenant:

  • Nom de la machine: www.iro.umontreal.ca
  • Chemin du fichier: ~vaucher/test.html
  • Protocole: http

Le navigateur convertit cela en une requete GET HTTP, envoyée au serveur avec l'adresse, le chemin, et d'autres informations (comme le port défaut: 80)

évolution vers le Web moderne

Web 3.0 et services en ligne

Ajourd'hui, les applications web vont au-delà de simples pages HTML

On parle d'applications conectées à des bases de données avec des services en ligne: Banques, e-commerce, météo, réservations..

Ces applications:

  • ont une interface fonctionnelle (frontend)
  • sont reliées à une base de données
  • offrent une interface d'administration(backend)

Pourquoi httpd seul ne suffit pas?

  • Les pages statistiqie ne suffisent plus
  • Il faut que la réponse du serveur soit dynamique (adaptée à l'utilisateur, etc)
  • Solution: ajout du CGI

CGI

CGI = Common Gateway Interface

CGI est une interface entre le serveur web (httpd) et des programmes externes

Cela permet de générer dynamiquement des pages

Par exemple:

Un utilisateur soumet un formulaire. Le serveur utilise CGI pour exécuter un script qui trtaite les données et retourne une page personnalisée

CGI a été une étape importante vers les applications web dynamiques

Application Web: Avantages, Inconvénient et Fondements

Avantages:

  • Portabilité de la partie client

    • Fonctionne sur tout navigateur (indépendamment du système d'explotation)

  • Déploiement facilité

    • Pas besoin d'installer l'application sur chaque poste; langages web sont standards

  • Accès massif

    • Pas de limitation sur le nombre d'utilisateurs simultanés

  • Sécurité simplifiée

    • Un seul protocole d'accés: HTTP ou HTTPS, via une URL unique

Inconvénient principal:

Difficile à tester et à mettre au point.

Pourquoi? En raison de l'interaction client-serveur asynchrone, des différences entre navigateurs, des dépendances réseau, etc
